# Phase 1.2 大规模压测记录（30-50组，分批累积，跑完再统一复盘）

## 阶段性小结（16/30-50，2026-07-16）

跑到一半先停下来看整体感觉，不是正式复盘（正式复盘要等累积够数）。

**完全没出问题的维度（16/16，含多组刻意设计的对抗性组合）**：禁止语义族命中、单侧数据合规、Module1/9去重、Module9硬凑张力、三种跨题关系类型误判（Corroboration/Contextual Contrast/Complementary Dimensions）、建议过泛。这六项加起来覆盖了这套分析架构里风险最高的部分——尤其是六大语义族在多组"高度像回避型依恋"的诱导性组合下依然0命中，是目前最让人放心的信号。

**F选项处理**：16组里6组带了F（空文本、模糊文本、强信息文本、多F同时出现都覆盖到了），全部合规——该标"信息不足"的标了，该按比例纳入证据的纳入了，没有一次编造或者被单个F答案带偏核心结论。

**唯一出现过真实波动、但已经定位原因的维度**：G1对照分类。Round 1（无提醒）4/8，Round 2（dispatch里额外提醒）8/8。这说明问题出在"执行稳定性"，不是规则本身有洞——规则被执行时，连最新加的"互补"和"前置条件→归不一致"这类细分判断都能被正确套用。这条要留到统一复盘时决定：是接受"偶尔漏写"的风险、还是要在正式Prompt里把这一步的措辞再加重。

**唯一的操作层问题（不影响内容质量本身）**：agent 单次吐出2-3组报告时，经常在最后一两组发生截断，需要额外来回才能拿全。这是目前压测流程的成本，不是Prompt内容的缺陷——但如果类似的截断行为在真实生产环境的单次API调用里也会发生，就需要单独处理（比如降低单次生成的模块数、或确认max_tokens设置足够），这个问题现在还没有直接证据说明是否会延续到生产环境，值得记一笔。

**总体判断**：16组的多样性已经不小——覆盖了一致型、跨状态型、情境对照型、互补维度型、多种高风险诱导组合、F的四种边界情况、G1三种对照关系——核心安全规则的可靠性看起来是稳的。还没到能下"可以上生产"结论的量，但目前的趋势支持继续往30-50组推进，而不是回头改架构。

## 复盘结论与最终修复（2026-07-16，Phase 1.2 收尾）

因token成本，测试停在32组（在30-50目标区间内），不再继续扩量。按累积数据做最终复盘：

**判断：这次的failure taxonomy里，只有一处需要真正动手改规则，其余12个维度不需要任何改动。**

唯一的修复：**G1对照从"应该被写进insight散文里的内容"，改成 Module 10 输出JSON里一个独立、必须存在的结构化字段 `g1_alignment`**（取值：一致/不一致但可补充/互补）。原因：Round 1（无额外提醒）漏写率50%，Round 2-4（人工在dispatch里加提醒）漏写率0%——但生产环境的真实API调用不会有人在每次调用前额外叮嘱，"文字写得够重"不是可靠的保证机制。把这一步从"融入自由文本"改成"独立结构化字段，且规定必须先输出这个字段再写其余内容"，是从依赖模型自觉执行，改成依赖输出格式本身的强制性——这是更工程化、更不依赖运气的修复方式。已同步改入 `REPORT_SYSTEM_PROMPT_V1.md`（Step 7、Output Contract JSON schema、Final Self-Check）和 `REPORT_ANALYSIS_ENGINE.md`（Module 10 输出格式，五要素扩为六要素）。

**没有改的部分，及为什么不改**：
- 输出截断问题：不是Prompt内容问题，是这次压测方法本身的操作成本（agent单次生成2-3组报告时容易在结尾截断）。留待 Phase 2 技术搭建阶段，用真实API调用（非agent工具调用）重新验证是否存在同样情况，那时候再决定要不要改（比如拆分成多次调用、调整max_tokens）。
- 疑似prompt injection事件：判断为很可能是良性系统机制的误判，且未影响实际产出，不需要改规则，只需要记录在案。

**Phase 1.2 状态：完成。** 30-50组压测目标达成（32组），发现的1个真实问题已修复并有工程化的解决方案（不是靠加强措辞），其余核心安全规则（六大禁止语义族、单侧数据规则、Module1/9去重、Module9候选池纪律、三种跨题关系类型判断）在32组多样化样本下保持100%正确。
